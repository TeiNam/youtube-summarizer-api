# 요구사항 문서

## 소개

YouTube Summary API에 API 인증키(API Key) 기반 인증 기능을 추가한다. 클라이언트는 API 키를 발급받아 모든 요청에 포함해야 하며, 유효한 API 키가 없는 요청은 거부된다. 기존 엔드포인트(`POST /summarize`, `GET /tasks/{task_id}`)에 인증을 적용하고, API 키의 발급 및 관리를 위한 엔드포인트를 제공한다.

## 용어 정의

- **API_서버**: FastAPI 기반의 유튜브 요약 REST API 서버
- **API_키**: 클라이언트를 식별하고 인증하기 위한 고유한 문자열 토큰
- **인증_미들웨어**: 모든 보호된 엔드포인트에 대해 API 키 유효성을 검증하는 미들웨어 컴포넌트
- **키_저장소**: API 키와 관련 메타데이터를 저장하고 관리하는 인메모리 저장소
- **관리자_키**: API 키 발급 및 폐기 등 관리 작업을 수행할 수 있는 환경변수 기반 마스터 키
- **보호된_엔드포인트**: 유효한 API 키가 필요한 API 엔드포인트 (`POST /summarize`, `GET /tasks/{task_id}`)
- **관리_엔드포인트**: API 키 발급, 목록 조회, 폐기를 수행하는 엔드포인트 (관리자_키 인증 필요)

## 요구사항

### 요구사항 1: API 키 발급

**사용자 스토리:** 관리자로서, 클라이언트에게 API 키를 발급하여 API 접근을 허용하고 싶다.

#### 인수 조건

1. WHEN 유효한 관리자_키와 클라이언트 이름이 포함된 POST 요청이 `/admin/api-keys` 엔드포인트로 전달되면, THE API_서버 SHALL 새로운 API_키를 생성하고 반환한다
2. THE API_서버 SHALL 발급된 API_키를 UUID v4 형식의 고유한 문자열로 생성한다
3. THE API_서버 SHALL 발급된 API_키와 함께 클라이언트 이름, 생성 일시, 활성 상태를 키_저장소에 저장한다
4. WHEN 관리자_키가 누락되거나 유효하지 않은 요청이 전달되면, THE API_서버 SHALL HTTP 403 상태 코드와 함께 권한 없음 메시지를 반환한다
5. WHEN 클라이언트 이름이 빈 문자열이거나 누락된 경우, THE API_서버 SHALL HTTP 422 상태 코드와 함께 필수 필드 누락 메시지를 반환한다

### 요구사항 2: API 키 인증

**사용자 스토리:** 사용자로서, 발급받은 API 키를 사용하여 API에 인증하고 싶다.

#### 인수 조건

1. THE 인증_미들웨어 SHALL 보호된_엔드포인트에 대한 모든 요청에서 `X-API-Key` 헤더의 존재 여부를 확인한다
2. WHEN 유효한 API_키가 `X-API-Key` 헤더에 포함된 요청이 전달되면, THE 인증_미들웨어 SHALL 해당 요청을 통과시킨다
3. WHEN `X-API-Key` 헤더가 누락된 요청이 보호된_엔드포인트로 전달되면, THE API_서버 SHALL HTTP 401 상태 코드와 함께 인증키 누락 메시지를 반환한다
4. WHEN 유효하지 않거나 존재하지 않는 API_키가 전달되면, THE API_서버 SHALL HTTP 401 상태 코드와 함께 유효하지 않은 인증키 메시지를 반환한다
5. WHEN 비활성화된 API_키가 전달되면, THE API_서버 SHALL HTTP 403 상태 코드와 함께 비활성화된 인증키 메시지를 반환한다

### 요구사항 3: API 키 관리

**사용자 스토리:** 관리자로서, 발급된 API 키를 조회하고 폐기할 수 있어야 한다.

#### 인수 조건

1. WHEN 유효한 관리자_키가 포함된 GET 요청이 `/admin/api-keys` 엔드포인트로 전달되면, THE API_서버 SHALL 발급된 모든 API_키의 목록을 반환한다
2. THE API_서버 SHALL API_키 목록 응답에 각 키의 클라이언트 이름, 생성 일시, 활성 상태를 포함한다
3. THE API_서버 SHALL API_키 목록 응답에서 API_키 값의 마지막 4자리만 노출하고 나머지는 마스킹 처리한다
4. WHEN 유효한 관리자_키가 포함된 DELETE 요청이 `/admin/api-keys/{key_id}` 엔드포인트로 전달되면, THE API_서버 SHALL 해당 API_키를 비활성화한다
5. IF 존재하지 않는 키 ID로 삭제 요청이 전달되면, THEN THE API_서버 SHALL HTTP 404 상태 코드와 함께 키를 찾을 수 없다는 메시지를 반환한다

### 요구사항 4: 관리자 인증

**사용자 스토리:** 운영자로서, 관리 기능에 대한 접근을 환경변수 기반 마스터 키로 보호하고 싶다.

#### 인수 조건

1. THE API_서버 SHALL 환경변수 `ADMIN_API_KEY`에서 관리자_키를 로드한다
2. WHEN 관리_엔드포인트에 대한 요청이 전달되면, THE API_서버 SHALL `X-Admin-Key` 헤더의 값을 관리자_키와 비교하여 검증한다
3. IF `ADMIN_API_KEY` 환경변수가 설정되지 않은 상태에서 API_서버가 시작되면, THEN THE API_서버 SHALL 경고 로그를 기록하고 관리_엔드포인트를 비활성화한다
4. THE API_서버 SHALL 관리자_키 비교 시 타이밍 공격(timing attack)을 방지하기 위해 상수 시간 비교(constant-time comparison)를 사용한다

### 요구사항 5: 기존 엔드포인트 인증 적용

**사용자 스토리:** 사용자로서, 기존 요약 API 엔드포인트에 인증이 적용되어 인가된 클라이언트만 사용할 수 있어야 한다.

#### 인수 조건

1. THE 인증_미들웨어 SHALL `POST /summarize` 엔드포인트에 API_키 인증을 적용한다
2. THE 인증_미들웨어 SHALL `GET /tasks/{task_id}` 엔드포인트에 API_키 인증을 적용한다
3. THE 인증_미들웨어 SHALL 헬스체크 등 공개 엔드포인트에는 인증을 적용하지 않는다
4. WHEN 인증에 실패한 요청이 전달되면, THE API_서버 SHALL 기존 비즈니스 로직을 실행하지 않고 즉시 오류 응답을 반환한다

### 요구사항 6: 오류 응답 형식

**사용자 스토리:** 사용자로서, 인증 관련 오류가 기존 API 오류 응답과 동일한 형식으로 반환되어야 한다.

#### 인수 조건

1. THE API_서버 SHALL 인증 오류 응답을 기존 `ErrorResponse` 형식(`error.code`, `error.message`)으로 반환한다
2. THE API_서버 SHALL 인증키 누락 오류에 `MISSING_API_KEY` 오류 코드를 사용한다
3. THE API_서버 SHALL 유효하지 않은 인증키 오류에 `INVALID_API_KEY` 오류 코드를 사용한다
4. THE API_서버 SHALL 비활성화된 인증키 오류에 `DISABLED_API_KEY` 오류 코드를 사용한다
5. THE API_서버 SHALL 관리자 권한 부족 오류에 `FORBIDDEN` 오류 코드를 사용한다

### 요구사항 7: 보안

**사용자 스토리:** 운영자로서, API 키 관리 시스템이 기본적인 보안 요건을 충족해야 한다.

#### 인수 조건

1. THE API_서버 SHALL API_키를 로그에 기록할 때 전체 키 값을 노출하지 않고 마스킹 처리한다
2. THE API_서버 SHALL 인증 실패 시도를 WARNING 레벨로 로그에 기록한다
3. THE 키_저장소 SHALL API_키를 해시(hash) 처리하여 저장하고, 원본 키는 발급 시에만 반환한다
4. THE API_서버 SHALL API_키 검증 시 해시된 값을 비교한다
